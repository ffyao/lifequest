# LifeQuest 开发流程

## 1. 流程总览

LifeQuest 采用“小组分工 + AI Coding + Gitee 分支协作 + 组长集成”的开发方式。

```text
选题与需求分析
→ 组长设计总体架构和模块边界
→ 成员在独立功能分支使用 AI Coding 开发
→ 成员推送分支到 Gitee
→ 组长合并到 dev 并处理冲突
→ 组长审查、测试、修复和补齐文档
→ 稳定版本合并到 main
→ 准备演示、截图和课程提交材料
```

## 3. 分支使用规范

### 3.1 分支说明

```text
main                  稳定展示分支
dev                   集成验证分支
integration/preview   合并预览分支
feature/frontend-ui   前端视觉与交互分支
feature/backend-crud  后端目标和任务 CRUD 分支
feature/ai-badges     AI Prompt、徽章文案和过程材料分支
feature/core          初始核心文档和协作说明分支
```

### 3.2 成员开发规范

成员在自己分支开发时遵守：

- 使用 SSH 连接 Gitee 仓库。
- 开发前先拉取最新代码，确认当前分支正确。
- 不直接修改 `main`。
- 一个提交尽量对应一个清晰模块。
- 不提交真实 DeepSeek API Key、Token、个人隐私数据、本地数据库或无关截图。
- 修改 API、数据库、Prompt 或核心规则时，同步补充说明或测试。

常用命令：

```bash
git clone git@gitee.com:tidehope/lifequest.git
cd lifequest
git checkout dev
git pull origin dev
git checkout -b feature/your-module
```

提交和推送：

```bash
git status --short
git add .
git commit -m "feat: describe your module"
git push origin feature/your-module
```

## 4. 组长集成流程

组长合并成员分支时按以下顺序进行：

```bash
git checkout dev
git pull origin dev
git fetch origin
git merge origin/feature/backend-crud
git merge origin/feature/ai-badges
git merge origin/feature/frontend-ui
```

如出现冲突：

1. 优先保留核心业务链路可运行。
2. 前端冲突以最终页面结构和样式一致为准。
3. 后端冲突以认证、权限、任务生成、任务完成和测试通过为准。
4. 文档冲突以当前实际实现为准。
5. 冲突解决后立即运行验证检查。

本项目实际集成记录包括：

```text
origin/feature/backend-crud → dev
origin/feature/ai-badges → dev
origin/feature/frontend-ui → dev
dev → main
```

## 5. 本地运行流程

安装和启动：

```bash
npm install
npm run dev
```

访问：

```text
http://localhost:3000
```

注意：本地服务是 HTTP，不要访问 `https://localhost:3000`。如果出现 `SSL_ERROR_RX_RECORD_TOO_LONG`，把地址改为 `http://localhost:3000`。

## 6. 验证检查流程

### 6.1 常规检查

每次合并或较大改动后执行：

```bash
node --check client/app.js
node --check server/services/api.js
node --check server/services/aiService.js
node --check server/services/database.js
node --check server/services/gameService.js
node --check server/services/goalService.js
node --check server/services/taskService.js
node --check server/services/userService.js
node --check server/tests/core-flow.test.js
git diff --check
npm test
```

### 6.2 手动演示检查

浏览器访问 `http://localhost:3000` 后检查：

```text
登录或注册
→ 管理员配置 DeepSeek API Key
→ 创建或查看角色
→ 点击头像上传图片
→ 输入目标并生成副本任务
→ 查看目标列表并切换目标
→ 删除目标并确认列表刷新
→ 完成主线、每日和 Boss 任务
→ 验证 XP、等级、徽章、排行榜更新
→ 验证目标通关后按钮显示“已通关”
```

## 7. 文档同步规则

| 改动类型 | 必须同步的文档 |
| --- | --- |
| API 路径、请求、响应、错误码 | `docs/API接口规范.md` |
| 数据库表、字段、核心数据流 | `docs/架构设计.md` |
| DeepSeek Prompt、模型、生成规则 | `docs/AI-Prompt设计.md` |
| 页面操作步骤、演示流程 | `docs/操作手册.md` |
| 功能范围和验收点 | `docs/规格说明.md` |
| 团队分工、分支协作、AI 使用过程 | `docs/协作文档.md` |
| 已发现问题和修复记录 | `docs/审查报告.md` |

## 8. 改动约束

- 认证、激活码、管理员权限不能绕过。
- 前端 UI 不展示默认账号密码、激活码、真实 API Key 或其他敏感信息。
- DeepSeek API Key 只由管理员配置，普通用户目标生成页不提供密钥输入框。
- 任务只能由 DeepSeek 生成，不使用本地模板兜底。
- 主线和 Boss 决定目标通关；每日任务每天可完成一次，但不决定通关。
- 主线、支线、Boss 任务不能重复获得 XP；每日任务同一天不能重复获得 XP。
- 用户名使用注册用户名，不提供修改入口；头像只能通过图片文件上传。
- 删除目标只删除目标本身，关联任务按数据库外键变为未归属目标，不直接删除任务记录。

## 11. 部署流程

### 11.1 本地演示部署

```bash
git clone git@gitee.com:tidehope/lifequest.git
cd lifequest
npm install
npm run dev
```

访问：

```text
http://localhost:3000
```

### 11.2 服务器部署

服务器要求：

```text
Node.js >= 22.5.0
npm >= 10
可写且可持久化的 server/data 目录
```

部署命令：

```bash
git clone git@gitee.com:tidehope/lifequest.git
cd lifequest
npm install
PORT=3000 npm start
```

后台运行：

```bash
nohup npm start > lifequest.log 2>&1 &
```

部署后处理：

- 使用 `http://服务器IP:3000` 访问。
- 如使用域名或 HTTPS，通过 Nginx 或平台网关反向代理到 Node.js 服务端口。
- 登录管理员账号，在“系统配置”页保存 DeepSeek API Key。
- 确认 `server/data/lifequest.sqlite` 能持久保存；不支持持久磁盘的平台需改用外部数据库。
