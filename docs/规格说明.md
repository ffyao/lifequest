# LifeQuest 人生副本任务系统规格说明

## 1. 项目背景

学生和职场人在学习、健身、备考、写作等长期目标中经常遇到两个问题：目标太大，不知道每天该做什么；普通待办工具缺少反馈和激励，容易半途而废。LifeQuest 将现实目标包装成 RPG 副本，通过任务、经验值、等级、徽章和排行榜提升用户持续行动的动力。

## 2. 项目目标

开发一个轻量全栈 Web 应用，让用户可以创建人生角色、输入目标，由 DeepSeek 自动生成主线任务、Boss 任务、可选每日任务和可选支线任务。用户完成任务后获得 XP、金币、等级提升和徽章奖励。

## 3. 目标用户

- 学生：课程学习、考试复习、技能提升。
- 健身人群：跑步、减脂、增肌、习惯养成。
- 创作者：写作、视频、自媒体、作品产出。
- 自律提升者：早睡、阅读、生活整理、时间管理。

## 4. 核心用户流程

```text
注册/登录
→ 创建人生角色
→ 输入目标
→ 生成副本任务
→ 查看目标列表并选择目标
→ 进入目标任务中心并完成任务
→ 获得 XP/金币
→ 等级提升
→ 解锁徽章
→ 查看排行榜
```

## 5. 功能需求

### 5.1 用户系统

- 用户可以注册账号。
- 用户可以登录账号。
- 登录后系统识别当前用户数据。

### 5.2 人生角色系统

- 用户可以创建或修改人生角色。
- 角色包含昵称、头像、职业标签、等级、XP、金币、连续打卡天数。
- 用户可以在副本看板点击头像选择图片文件作为头像。
- 用户名使用账号注册时的用户名，不提供用户名修改或个性签名机制。
- 完成任务后角色 XP 和等级会更新。

### 5.3 目标系统

- 用户可以创建长期目标或短期冲刺目标。
- 目标包含标题、描述、分类和状态。
- 目标列表按创建时间倒序展示，最新目标在最上方。
- 系统会记住当前用户上次选择的目标，刷新后默认恢复对应任务中心。
- 用户可以在目标列表中删除自己的目标。
- 目标分类包括学习、健身、阅读、创作、生活。

### 5.4 副本任务系统

- 系统可以根据目标生成任务。
- 任务类型包括主线任务、Boss 任务、可选每日任务和可选支线任务。
- 任务包含标题、描述、类型、难度、XP 奖励和完成状态。
- 用户可以在选中目标的任务中心完成任务。
- 主线、支线和 Boss 任务完成后不可重复获得 XP。
- 每日任务适用于长期目标或习惯养成目标，每天可以完成一次；短期目标、一次性目标或冲刺型目标允许不生成每日任务。
- 每日任务当天完成后显示“今日已完成”，次日可再次挑战。
- 目标通关进度只取决于主线任务和 Boss 任务；主线和 Boss 全部完成后目标视为通关。
- 目标通关后，该目标下所有任务的完成按钮统一显示“已通关”。

### 5.5 AI 任务生成系统

- 系统通过 DeepSeek `deepseek-v4-flash` 模型根据用户目标生成结构化任务。
- DeepSeek API Key 由管理员统一配置，普通用户无需填写密钥即可生成任务。
- 任务只能由 AI 生成，后端不使用本地模板兜底。
- 生成结果必须包含 3 到 6 个任务，且至少包含 1 个主线和 1 个 Boss；每日任务和支线任务可选。
- 模型必须为每个任务生成 `xpReward`，每日任务经验通常低于同等投入的主线或支线任务，但具体数值由模型根据任务行动成本决定。
- 后端会校验模型返回的 JSON、任务类型、任务难度、XP 整数范围、标题和描述，校验通过后才写入数据库。
- 系统保存生成日志，便于展示 AI 辅助过程。

### 5.6 等级与徽章系统

- 完成任务获得 XP。
- XP 每满 100 提升 1 级。
- 达成条件后自动解锁徽章。
- 徽章包括首次启程、目标制定者、副本挑战者、连击达人等。

### 5.7 排行榜系统

- 系统按 XP 和等级展示用户排名。
- 排行榜展示用户当前头像和账号注册用户名。
- 排行榜用于小组展示和游戏化激励。

## 6. 非功能需求

### 6.1 性能

- 本地运行时主要页面响应应在 1 秒以内。
- 常用接口应返回轻量 JSON 数据。

### 6.2 兼容性

- 支持现代浏览器，如 Chrome、Edge。
- 后端运行环境为 Node.js 22 及以上。

### 6.3 安全性

- 演示版本不保存真实隐私数据。
- 密码仅用于课程原型演示，正式版本应使用哈希存储。
- API 使用登录后返回的 Bearer token 访问业务接口。
- 登录或注册成功后会话有效期为 7 天，用户可以主动退出登录。
- 注册账号必须填写有效激活码，普通激活码一次性使用，高级激活码支持多次激活。
- DeepSeek API Key 只用于服务端调用模型，由管理员统一配置；前端只显示是否已配置，不展示密钥明文。

### 6.4 可维护性

- 前端、后端、文档分目录管理。
- 核心规则集中在后端服务层。
- AI 生成逻辑与任务逻辑分离，便于替换真实模型。

## 7. 需求优先级

### Must Have

- 注册登录。
- 角色创建与展示。
- 目标创建。
- 任务生成。
- 目标列表和目标任务中心。
- 任务完成。
- XP 和等级更新。
- 本地可运行。

### Should Have

- 徽章墙。
- 排行榜。
- NPC 鼓励语。
- AI 生成日志。
- 游戏化首页和副本看板。

### Nice to Have

- 小组共享副本和多人任务。
- 数据统计图表。
- 成就分享海报。
- 日历视图。

## 8. 范围控制

本项目不做复杂社交系统、支付系统、多角色权限系统和真实移动端小程序。课程提交阶段优先保证核心链路完整可演示，再逐步增加视觉包装和 AI 扩展。
