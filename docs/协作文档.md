| AI Coding 工具 | 需求拆解、代码生成、代码审查和文档整理 | 组长给成员 AI 提供完整任务提示词，成员 AI 在终端和文件权限内完成对应模块 |
| DeepSeek | 产品侧任务生成大模型 | 系统运行时由后端调用 `deepseek-v4-flash` 生成目标任务，不在前端暴露 API Key |
| npm scripts | 本地运行与验证 | 使用 `npm run dev` 启动演示，使用 `npm test` 验证核心业务流程 |
| Markdown 文档 | 课程材料沉淀 | 用 `docs/` 目录维护规格、架构、接口、操作、审查、Prompt 和协作记录 |

协作约定：

- 成员 AI 使用 SSH 地址 `git@gitee.com:tidehope/lifequest.git` 克隆仓库，不使用 HTTPS 密码方式。
- 成员 AI 只修改自己负责的模块，不回退其他成员或组长的提交。
- 推送前至少运行与自己模块相关的本地检查；如果无法运行，需要在提交说明中写明原因。
- 真实 DeepSeek API Key、Token、个人隐私数据和本地数据库文件不得提交到仓库。

## 5. AI Coding 协作方式

本项目的开发特点是“人负责目标和判断，AI 负责辅助实现”：

```text
组长拆解需求
→ 为成员划分模块和分支
→ 成员使用 AI Coding 工具在功能分支开发
→ 成员推送到 Gitee
→ 组长拉取并合并到 dev
→ 组长审查、测试、修复、统一文档
→ 稳定后合并到 main
```

AI 主要参与以下工作：

- 需求分析：比较多个选题，最终确定 LifeQuest 方向。
- 架构设计：拆分前端、后端、数据库、AI 服务和文档模块。
- 代码生成：生成页面、接口、服务层、测试和样式。
- 代码审查：检查认证、权限、任务完成、XP、目标通关和数据一致性。
- 文档整理：生成和同步 README、API 规范、架构说明、操作手册和 Prompt 文档。

### 5.2 AI 输出确认方式

- 需求阶段：组长比较 AI 给出的多个选题后，人工选择“人生副本任务系统”作为最终方向。
- 设计阶段：AI 给出技术栈和模块拆分建议，组长根据小组能力确认采用原生前端、Node.js 和 SQLite。
- 开发阶段：成员 AI 生成初版代码，成员与组长通过运行结果、接口返回和测试用例判断是否可合并。
- 审查阶段：组长使用 AI 辅助检查登录鉴权、权限隔离、AI 任务生成、目标删除和通关逻辑。
- 文档阶段：AI 辅助整理文档，组长按课程评分标准逐项核对，删除不准确或泄密内容。

## 6. 产品侧 AI 功能

LifeQuest 的产品侧 AI 功能是“人生副本任务生成”：

```text
用户输入目标
→ 管理员预先配置全局 DeepSeek API Key
→ 后端调用 DeepSeek deepseek-v4-flash
→ DeepSeek 返回结构化任务 JSON
→ 后端校验任务数量、类型、难度、XP 范围、文本长度和模板化标题
→ 写入任务表和 AI 日志
→ 前端展示主线、Boss、可选每日和可选支线任务
```

关键约束：

- 任务只能由 DeepSeek 生成，不使用本地模板兜底。
- 每次生成必须得到 3 到 6 个任务。
- 类型组合至少包含 1 个主线和 1 个 Boss；每日任务和支线任务可选。
- 短期目标、一次性目标或冲刺型目标允许不生成每日任务；长期目标和习惯养成目标可以生成每日任务。
- Boss 任务难度必须为 `boss`。
- 每日任务每天可完成一次，不决定目标通关。
- 目标通关只取决于主线和 Boss，全部完成后目标下所有按钮显示“已通关”。
- XP 由模型生成 `xpReward`，后端校验为 5 到 220 的整数；每日任务经验通常偏低，但不使用固定难度映射。
- 可修复的模型格式问题会携带失败原因重试一次。
- 生成失败不写入任务，也不回退到本地模板。

## 7. 集成与质量控制

组长集成成员分支时采用以下标准：

- **能运行**：`npm run dev` 后可访问 `http://localhost:3000`。
- **能测试**：`npm test` 覆盖注册登录、激活码、头像、AI 任务生成、任务完成、目标删除、任务编辑和权限隔离。
- **能演示**：登录、生成任务、完成任务、查看徽章和排行榜的主链路不能中断。
- **不泄密**：不提交真实 DeepSeek API Key、Token、个人隐私数据或本地数据库文件。
- **文档同步**：API、数据库、Prompt、操作步骤变化后同步更新文档。

## 8. 协作效率对比

| 环节 | 纯手写方式 | AI 辅助方式 | 实际效果 |
| --- | --- | --- | --- |
| 选题与需求拆解 | 需要人工反复整理功能边界，容易遗漏非功能需求 | AI 快速列出多个方案和功能清单，组长再筛选可控范围 | 更快形成规格说明，减少范围失控 |
| 前端页面开发 | 逐页手写 HTML/CSS/JS，样式调整耗时 | AI 生成布局初稿，成员再按演示效果调整 | 更快得到可演示界面，但仍需人工统一视觉和交互 |
| 后端接口开发 | 手动编写接口、校验和测试，容易漏权限 | AI 根据 API 规范生成接口与测试思路 | 提高 CRUD 开发速度，但权限隔离必须重点复查 |
| AI Prompt 设计 | 需要多次试错才能稳定输出结构化任务 | AI 辅助设计 JSON Schema、失败重试和反模板化约束 | 输出更接近产品需求，但仍需后端强校验 |
| 代码审查 | 依赖人工逐文件阅读，容易忽略边界条件 | AI 按模块检查认证、数据一致性和异常分支 | 更容易发现潜在问题，最终决策仍由组长确认 |
| 文档整理 | 手动维护多份文档，容易和代码不同步 | AI 根据代码和评分标准生成文档框架 | 提升完整性，但需要人工核对真实性和敏感信息 |

总体来看，AI 降低了从想法到可运行版本的时间成本，但不能替代人工判断。尤其是登录权限、API Key 安全、数据库写入、目标通关规则等关键逻辑，必须通过测试和人工审查确认。

## 10. 经验总结与反思

本项目体现了小组协作和 AI Coding 的结合：组员在独立分支完成前端、后端和 AI 文案相关模块，组长负责架构、集成、质量控制和最终稳定化。最终版本保留了徐丙伦、黎仲春、尹严友在不同模块中的贡献，同时通过统一测试和文档整理保证项目可以稳定演示。

经验总结：

- 任务拆分要先明确“模块边界”和“不可破坏的核心流程”，否则多个 AI 同时开发时容易出现接口不一致。
- 给成员 AI 的提示词必须包含仓库地址、分支、阅读文档、修改范围、测试要求和推送规范，不能只写一句功能需求。
- AI 生成的代码需要重点审查安全和数据一致性，特别是登录会话、激活码、API Key、用户权限和数据库写入。
- 文档应随着代码同步更新，最终提交前再按评分标准逐项核对，保证规格、架构、审查、操作和协作材料齐全。
- AI 可以显著提高开发速度，但组长仍需要承担产品取舍、架构统一、冲突处理、验收测试和最终质量负责。

后续改进方向：

- 将 SQLite 升级为更适合多人部署的外部数据库。
- 增加更完整的前端自动化测试，覆盖登录、生成目标、完成任务和排行榜展示。
- 对密码存储使用更强的哈希算法，并补充 HTTPS 生产部署说明。
- 将 AI 任务生成的日志可视化，方便展示 Prompt 迭代和模型输出质量。
